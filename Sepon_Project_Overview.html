01

ที่มา: ทำไมต้องเริ่มโครงการนี้

NB หัวหน้าบัญชี เห็นปัญหาจากการนั่งทำงบต้นทุนเอง

ปัญหาราก

งบต้นทุนเดิมเป็นการ “รวมเลขทางบัญชี” ไม่ได้แตกส่วนของต้นทุนออกมา

พอต้นทุนผิดปกติ ก็หาสาเหตุไม่ได้ — และระบบหน้างานเดิมยังไม่พร้อมป้อนข้อมูลที่ดีพอสำหรับการทำต้นทุน

ของไม่ผ่านคลัง

ซื้อแล้วเข้าหน้างานเลย — ยอดซื้อกับยอดเบิกต่างกันมาก ตามรอยการใช้จริงไม่ได้

ข้อมูลออกช้า

ต้องกรอกมือหลายชั้น กว่าจะรวมเป็นงบได้ก็ช้าเกินใช้บริหารหน้างาน

วิเคราะห์ไม่ได้

ไม่รู้ว่าต้นทุนก้อนไหนไปกับงานเจาะ/ระเบิด/ขุดขน — วัดประสิทธิภาพต่อ BCM ไม่ได้

02

เป้าหมายของโครงการ

จัดการข้อมูลให้ครบก่อน แล้วต้นทุนที่ออกมาถึงจะสมบูรณ์พอไปวิเคราะห์ต่อ

ข้อมูลครบถ้วน

ดันทุกการซื้อ–รับ–เบิก ให้เข้าระบบ 100% ไม่ตกหล่น

เร็วใกล้ Real-time

เห็นต้นทุนทันทีที่เกิดรายการ ไม่ต้องรอปิดงบ

เป็น Digital ทั้งหมด

เลิกกรอกมือ ใช้แสกน + ลิงก์ข้อมูลอัตโนมัติ

ผลที่ต้องการ

ต้นทุนที่ “เล่าเรื่องในโครงการได้” — ออกเร็วที่สุด ถูกต้องสมบูรณ์พอจะวิเคราะห์และบริหารหน้างานได้ถูกทิศ

05

งานที่ 1 — จัดซื้อ: ติดตามงาน กันออเดอร์หลุด

ยังไม่แตะ flow งาน แค่ทำให้ปัญหาเบาลงด้วยข้อมูลที่ถูกต้องขึ้น

ปัญหา: งานพอกหางหมู
10 → 20 → 30+

PR เข้าช่วงพีค >10/วัน ทำเสร็จยังต้องตามจ่ายต่อ แต่ติดสภาพคล่อง งานเลยสะสม

ผลกระทบ: หลายรายการที่ต้องซื้อถูกลืม/หลุด กระทบหน้างาน — เรื้อรังแม้เปลี่ยน/เพิ่มคนแล้ว

เว็บติดตามงานจัดซื้อ (pr-warehouse)

เลิกกรอก Google Sheet มือ → ดึงจาก TRCloud โดยตรงด้วย web scraping ลิงก์ real-time กับฐานข้อมูลบริษัท ลดขั้นตอน + ข้อมูลถูกต้องขึ้น (เป็นเอกสารที่ผ่านหัวหน้าตรวจแล้ว)

จุดเด่น: ปรับฟีเจอร์ตามหน้างานได้ภายในวัน (< 15 นาที)

GitHubวางโค้ด
VS Codeเขียน/แก้
AI / Claudeแก้ด้วย prompt
ใช้งานจริงdeploy ทันที
06

งานที่ 2 — คลังน้ำมัน: รายงานสด + ต้นทุนรายวัน

ยุบหลายชีทเป็นชีทเดียว ลิงก์จาก MES → แสดงสถานะใกล้ real-time

เว็บรายงานน้ำมัน (Fuel2)

เดิมทีมน้ำมันกรอกหลายชีทเพื่อรายงานรายวัน + ประชุม weekly → ยุบเป็นชีทเดียว ดึงจาก MES ตรง เป็นข้อมูลตัวเดียวกับงบสะพาน

ต้นทุนน้ำมัน FIFO (MWA)

ตีราคาน้ำมันที่เบิกแบบ moving-weighted-average ระบุต้นทุนจริงระดับรายวัน — กลั่นเป็น skill ใช้ซ้ำได้ทุกเดือน

แผนแอปแสกนจ่ายน้ำมัน — ลดจุดแสกน

ปัจจุบัน 4 จุด: หัวจ่าย · คนจ่าย · รถ · คน (พนักงานลืมพกบัตร = เติมไม่ได้ → ถอยไปเขียนมือ)

4 → 2
เหลือแค่ หัวจ่าย + รถ
คนจ่าย — จาก login (แอป offline → push เมื่อมีเน็ต)
คนขับ — จากแถบแม่เหล็ก/GPS ที่ใช้สตาร์ทรถ (ชั่วคราวแสกนบัตร LXML)
ไม่มีบัตร — เติมได้ก่อน ไม่บล็อก production — บันทึกไว้จัดการทีหลัง
07

งานที่ 3 — คลังพัสดุ: หัวใจของต้นทุน

ดันของให้ผ่านคลัง 100% เพื่อผูกการเบิกเข้ากับรถที่ซ่อมจริง → ต้นทุนต่อ BCM

ปัญหา

ยอดซื้อ ≠ ยอดเบิก

ของส่วนใหญ่ไม่ผ่านคลัง — ซื้อแล้วเข้าหน้างานเลย หรือเขียนใบเบิกแต่ไม่ลงข้อมูล จึงไม่รู้ว่าใช้จริงเมื่อไหร่ กับรถคันไหน

ผูก Work Order

ทุกการเบิกผูก รถ + cost center + กิจกรรม (air code ฝังในฐานข้อมูล)

ต้นทุนต่อรถ/งาน

DB function สรุปต้นทุน ABC รายรถ + ราย cost center อัตโนมัติ

Handshake 2 ฝั่ง

แสกนยืนยันคนเบิก–คนจ่าย · ระบบจับยืนยันข้างเดียวทุกชั่วโมง

Soft-block

ของด่วนเบิกผ่านได้ ไม่บล็อก production แต่ติดธงไว้ตรวจ

Offline-first

บันทึก SQLite ในเครื่อง → sync ขึ้น cloud เมื่อมีเน็ต (เน็ตหน้างานแย่)

Cycle count

กระทบยอดช่องว่างซื้อ–เบิก ระบุเหตุ เช่น เบิกไม่ลงบันทึก

08

สถาปัตยกรรมคลัง: Offline-first → Cloud → ERP

หน้างานไม่มีวันสะดุดเพราะเน็ต — ข้อมูลพักในเครื่องก่อน แล้วค่อย sync

อุปกรณ์หน้างาน

Tablet/Mobile (Expo) · กรอก/เลือก/ถ่ายรูป/แสกน

เก็บในเครื่อง

SQLite + sync queue · status: pending (ออฟไลน์)

ฐานข้อมูลกลาง

Google Cloud · PostgreSQL + realtime

TRCloud ERP

POST รับเข้า (GR) / จ่ายออก (GI) · เอกสาร + สต็อก

Sync เช็คสองทาง (two-way)

เทียบข้อมูลทั้งสองฝั่ง ถ้าไม่ตรง เติมส่วนที่หายไป · ใช้ offline_id เป็น idempotency key กันบันทึกซ้ำ · ข้อมูลแบบ change-on-edit · ตัดสต็อกแสดงผลบนเว็บ/แอปแบบ real-time

09

งานที่ 4 — คลังโรงครัว: ต้นทุนต่อหัว/วัน

ไม่ลงลึกเท่าเหมือง/คลังพัสดุ — เน้นแสดงต้นทุน real-time ทันทีที่รับเข้า/เบิก

11

สถานะ & ก้าวต่อไป

ทำแล้ว

งบสะพาน (P&L) ออกได้ทุกเดือน
จัดซื้อ — เว็บติดตามงาน pr-warehouse
คลังน้ำมัน — เว็บรายงาน + ต้นทุน FIFO (skill)
คลังพัสดุ — แอป offline-first สร้างจริง (schema + DB function)
คลังโรงครัว — ออกแบบระบบ + โครงต้นทุน

ก้าวต่อไป

นำแอปคลังพัสดุขึ้นใช้จริง — ของผ่านคลัง 100%
แอปแสกนจ่ายน้ำมัน (ลด 4→2 จุด) + GPS รถ
คลังโรงครัว — ตัดสต็อก + ต้นทุนต่อหัวสด
เชื่อมทุกคลังเข้างบต้นทุน → ต้นทุนต่อ BCM real-time
ลดการพึ่ง Estimation ในงบสะพานทีละส่วน
